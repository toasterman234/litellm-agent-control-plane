## Components

| Component | Where it runs | What it does |
|---|---|---|
| Web and API service | `lite` on `:4000` | Serves the dashboard, REST API, gateway routes, and session streams |
| Postgres | Docker Compose or your database | Stores agents, sessions, runtime harnesses, credentials, schedules, memory, and settings |
| LiteLLM gateway | Rust service | Routes model calls, authenticates requests, proxies MCP traffic, and calls upstream providers |
| Runtime harnesses | Built-in providers or template services | Run agent sessions through Claude Managed Agents, Cursor, OpenCode, DeepAgents, Hermes, or Elastic Runtime |
| Runtime adapters | `src/sdk/providers/<provider>/runtime/` | Normalize each runtime to the same managed-agent session and event shape |
| UI channels | Dashboard, API, Slack | Give teams shared surfaces for creating agents and running sessions |

## Request flow

1. You create an agent in the UI or API.
2. The platform stores the agent, tools, MCP servers, model settings, and runtime selection in Postgres.
3. You start a session from the UI, API, or Slack.
4. The session resolves a runtime alias, loads the right credential, and calls the runtime adapter.
5. The selected runtime creates or resumes the remote session.
6. Events stream back through LAP as a normalized session stream.
7. The dashboard and API clients render the same session state.

## Local stack

Docker Compose starts the web/API service, Postgres, and optional template runtime profiles. Runtime profiles register themselves through the LAP API after the service is healthy, so `local-opencode`, `local-deepagents`, and `local-hermes` appear in the UI automatically.
